01 — Context

Enterprise Lending at Scale

Digital lending platforms sit at the intersection of financial regulation, workflow orchestration, and real-time decisioning. This project (VelocityCredit) was built as a full-stack major project to simulate a production-grade enterprise lending system — API-first, with clear domain boundaries and a clean architecture.

The platform was designed to handle the core lending lifecycle: application intake, AI-assisted credit evaluation, loan workflow management, disbursement tracking, and analytics — with a reference C#/.NET implementation.

02 — Problem

What Enterprise Lending Actually Requires

  • Credit decisioning must be rule-based, auditable, and explainable — not a black box
  • Lending workflows have complex state machines: applied → verified → approved → disbursed → repaying → closed
  • APIs must be versioned and contract-stable — downstream integrations break on schema changes
  • The system must handle concurrent loan applications without race conditions on shared credit limits
03 — Architecture

System Structure

VelocityCredit.sln (ASP.NET Solution) │ ├── src/ │ ├── VelocityCredit.API/ ← Controllers, Middleware, DI │ ├── VelocityCredit.Core/ ← Domain models, interfaces │ ├── VelocityCredit.Application/ ← Business logic, CQRS handlers │ ├── VelocityCredit.Infrastructure/ ← EF Core, SQL Server, repos │ └── VelocityCredit.AI/ ← Credit scoring, rule engine │ ├── tests/ │ ├── Unit/ ← Domain logic tests │ └── Integration/ ← API + DB integration tests │ └── docker/ ← Compose for local dev stack

The solution follows Clean Architecture — the domain layer has zero external dependencies. Business logic lives in Application, and the API layer is a thin controller surface over application services.

04 — Key Decisions

Design Choices

  • C#/.NET for the core: The .NET ecosystem is the industry standard for enterprise financial applications — strong typing, mature ORM (EF Core), and excellent testing tooling
  • CQRS pattern: Commands (mutations) and queries are separated, making the credit decisioning logic independently testable and auditable
  • Rule-based credit scoring: Implemented as a configurable rule engine rather than an ML model — easier to audit, explain to regulators, and update without retraining
  • API-first design: All functionality exposed via RESTful endpoints with OpenAPI documentation — documented in API_REFERENCE.md (15KB)
  • Docker Compose for local dev: Ensures the full SQL Server + API stack runs identically on any machine
